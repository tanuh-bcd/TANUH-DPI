Setup Steps (All Platforms)

1. Install Ollama

Download from ollama.com/download
Linux: curl -fsSL https://ollama.com/install.sh | sh

2. Pull the LLM Model (~16 GB download)

ollama pull gemma4:26b
Requires NVIDIA GPU with 16 GB+ VRAM, or Apple Silicon with 16 GB+ unified memory.

3. Unzip & Make Executable

Unzip the downloaded file. On Linux/Mac run: chmod +x nhcx-extract*
Linux requires sudo apt install libfuse2 if not already installed.

4. Run NHCX Insurance Extraction

./nhcx-extract nhcx policy.pdf -o result.json -m gemma4:26b

Use check first to verify setup: ./nhcx-extract check -m gemma4:26b

Privacy: All data stays on your machine. Zero cloud API calls. The output JSON contains a complete FHIR R4 Bundle.

 Overview

    
DPI Tanuh AI is a Digital Public Infrastructure platform developed by Tanuh AI-Center of Excellence at IISc Bangalore. The pdf2nhcx service automates the conversion of insurance documents (health insurance policies, claims, pre-authorization forms) into machine-readable FHIR R4 InsurancePlanBundles compliant with India's National Health Claims Exchange (NHCX) standards.

    

        
FeatureDescription

        
            
Document IntelligenceAutomatic classification of insurance documents with two-tier validation gate

            
Single Bundle OutputAll insurance document types converted into a unified InsurancePlanBundle format

            
FHIR R4 ComplianceGenerated bundles conform to NHCX StructureDefinition profiles

            
LangGraph OrchestrationDependency-aware extraction using topological sort for correct resource ordering

            
Async ProcessingCelery + Redis task queue supports up to 12 concurrent PDF processing slots

 NHCX Insurance Pipeline

    
The pdf2nhcx service converts insurance documents into FHIR R4 InsurancePlanBundle resources conforming to National Health Claims Exchange standards. All insurance document types are converted into this single bundle format.


    
Input & Output

    
Accepts any insurance-related PDF document (health insurance policies, claims, etc.). Output is always a single InsurancePlanBundle — a FHIR Bundle of type collection containing InsurancePlan, Organization, Condition, DocumentReference, and other relevant resources extracted from the document.


    
Resource Dependency Graph

    

        Organization → InsurancePlan → InsurancePlanBundle

        Independent: Binary, DocumentReference, Condition, Patient, Practitioner
    


    
Pipeline Steps

    

        

            1. OCR + Distillation
            
Multi-engine OCR extracts raw text, then 8 LLM calls distill structured information from the entire text.

        

        

            2. Document Type Gate
            
Classifier validates the document is an insurance document. Clinical docs are rejected with a redirect message.

        

        

            3. NHCX Resource Selection
            
LLM identifies which insurance-specific FHIR resources are present in the document.

        

        

            4. LangGraph Extraction
            
Each resource extracted via dedicated LLM call with NHCX rulebook (StructureDefinition JSON) loaded into prompt context.

        

        

            5. Sanitization & Assembly
            
sanitize_fhir_resource() fixes hallucinated types, missing required fields, invalid code systems. Bundle assembled with InsurancePlan first.

        

        

            6. PDF Embedding & Upload
            
Original PDF embedded as base64 Binary resource. Final bundle uploaded to GCS and cached in Redis (24h TTL).

        

    


    
Rulebook-Driven Extraction

    
Each FHIR resource type has a corresponding StructureDefinition JSON file (40+ rulebooks in pdf2nhcx/rulebooks_updated/) loaded into the LLM prompt. This ensures generated resources conform to the exact field constraints, cardinality rules, and terminology bindings defined by NHCX.

    

        Cross-Reference Integrity: An id_registry dictionary tracks generated resource IDs across the workflow, ensuring that references (e.g., InsurancePlan referencing Organization) use correct UUIDs.
    







    
 OCR Engine Waterfall

    
6-engine waterfall strategy to maximize text extraction quality across diverse PDF types (digital, scanned, handwritten, complex layouts).

    

        
#EngineTechnologyBest ForFallback Trigger

        
            
1PyPDFNative PDF text layerClean digital PDFs<50 chars/page (scanned PDF)

            
2Chandra2VLM 4B (datalab/chandra2)Complex layouts, handwritingGPU unavailable or model error

            
3DoclingIBM Structural ParserTables, structured documentsParse failure or empty output

            
4LightOnPyMuPDF4LLMFast digital extractionEmpty or insufficient text

            
5SuryaTransformer OCR90+ languages, 2x resolutionLow confidence scores

            
6ChandraTesseract (300 DPI)Last resort, any documentFinal fallback (always succeeds)

        
    

    

        GPU Optimization: Chandra2 (VLM) requires 8-12GB VRAM. When no GPU is available, the waterfall automatically skips to Docling, maintaining service availability without hardware dependency. All engines produce uniform Markdown output through a common normalizer.

 Validation & Sanitization

    
LLM-generated FHIR resources undergo multi-layer validation and correction via sanitize_fhir_resource() before inclusion in the final bundle.

    

        
IssueFix Applied

        
            
Wrong resourceTypeCorrected to match expected type

            
Missing required fieldsPopulated with sensible defaults

            
Invalid code systemsReplaced with correct canonical URLs

            
Broken internal referencesResolved using ID registry

            
Incorrect date formatsNormalized to FHIR dateTime format

            
Hallucinated extensionsRemoved if not in profile

        
    

    
40+ StructureDefinition JSON rulebooks in pdf2nhcx/rulebooks_updated/ specify required vs. optional fields (cardinality), allowed value sets and code systems, data type constraints, and slicing rules for every supported NHCX resource type.
