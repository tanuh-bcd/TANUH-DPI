Setup Steps (All Platforms)

1. Install Ollama

Download from ollama.com/download
Linux: curl -fsSL https://ollama.com/install.sh | sh

2. Pull the LLM Model (~16 GB download)

ollama pull gemma4:26b
Requires NVIDIA GPU with 16 GB+ VRAM, or Apple Silicon with 16 GB+ unified memory.

3. Unzip & Make Executable

Unzip the downloaded file. On Linux/Mac run: chmod +x nhcx-extract*
Linux requires sudo apt install libfuse2 if not already installed.

4. Run NHCX Insurance Extraction

./nhcx-extract nhcx policy.pdf -o result.json -m gemma4:26b

Use check first to verify setup: ./nhcx-extract check -m gemma4:26b

Privacy: All data stays on your machine. Zero cloud API calls. The output JSON contains a complete FHIR R4 Bundle.

How to Use

            
Follow these three steps to convert an insurance PDF into an NHCX-compliant FHIR bundle.


            

                

                    
1. Upload PDF

                    
Click "Choose PDF file..." and select an insurance document (health insurance policy, pre-authorization, or claims form). Maximum file size is 25 MB.

                

                

                    
2. Process Document

                    
Click "Process insurance document". The AI pipeline extracts text via OCR, validates the document type, and generates FHIR resources. Processing takes 3–5 minutes.

                

                

                    
3. Download FHIR Bundle

                    
Once complete, the NHCX-compliant FHIR JSON appears in the output panel. Use Copy JSON or Download JSON to save. Validate using the Validate FHIR R4 button.

NHCX Insurance Pipeline

            
The pdf2nhcx service converts insurance documents into FHIR R4 InsurancePlanBundle resources conforming to National Health Claims Exchange standards. All insurance document types are converted into this single bundle format.


            
Resource Dependency Graph

            
NHCX resources have strict ordering requirements. The pipeline uses topological sorting to ensure dependencies are resolved before dependent resources are extracted:

            
  Organization ──────────┐
                         v
  InsurancePlan ─────────┤
                         v
  InsurancePlanBundle ───┘

  Independent: Binary, DocumentReference, Condition, Patient, Practitioner


            
Pipeline Steps

            

                
1. OCR + Distillation
Multi-engine OCR extracts raw text, then 8 LLM calls distill structured information from the entire text.

                
2. Document Type Gate
Classifier validates the document is actually an insurance document. Clinical docs are rejected with a redirect message.

                
3. NHCX Resource Selection
LLM identifies which insurance-specific FHIR resources are present in the document.

                
4. LangGraph Extraction (Dependency-Ordered)
Each resource extracted via dedicated LLM call with NHCX rulebook (StructureDefinition JSON) loaded into prompt context.

                
5. Sanitization & Assembly
sanitize_fhir_resource() fixes hallucinated types, missing required fields, invalid code systems. Bundle assembled with InsurancePlan first.

                
6. PDF Embedding & Caching
Original PDF embedded as base64 Binary resource. Final bundle cached in Redis (24h TTL).

            


            
Rulebook-Driven Extraction

            
Each FHIR resource type has a corresponding StructureDefinition JSON file (40+ rulebooks) loaded into the LLM prompt. This ensures generated resources conform to the exact field constraints, cardinality rules, and terminology bindings defined by NHCX.


            

                Cross-Reference Integrity
                An id_registry dictionary tracks generated resource IDs across the workflow, ensuring that references (e.g., InsurancePlan referencing Organization) use correct UUIDs.
            

        


        
        

            
OCR Engine Waterfall

            
The platform employs a 6-engine waterfall strategy to maximize text extraction quality across diverse PDF types.


            

                
#EngineTechnologyBest ForFallback Trigger

                
                    
1PyPDFNative PDF text layerClean digital PDFs<50 chars/page (scanned)

                    
2Chandra2VLM 4B (datalab/chandra2)Complex layouts, handwritingGPU unavailable or error

                    
3DoclingIBM Structural ParserTables, structured docsParse failure or empty

                    
4LightOnPyMuPDF4LLMFast digital extractionEmpty or insufficient text

                    
5SuryaTransformer OCR90+ languages, 2x resolutionLow confidence scores

                    
6ChandraTesseract (300 DPI)Last resort, any documentFinal fallback (always succeeds)

                
            


            

                GPU Optimization
                Chandra2 (VLM) requires 8–12 GB VRAM. When no GPU is available, the waterfall automatically skips to Docling, maintaining service availability without hardware dependency.

Quick Start (curl)

            
# 1. Get a token
TOKEN=$(curl -s -X POST https://dpi-dev.tanuh.ai/pdf2nhcx/api/token \
  -H "Content-Type: application/json" \
  -d '{"name":"Your Name","email":"you@example.com"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

# 2. Synchronous conversion
curl -X POST https://dpi-dev.tanuh.ai/pdf2nhcx \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@insurance_policy.pdf"

# 3. Asynchronous submission
curl -X POST https://dpi-dev.tanuh.ai/pdf2nhcx/submit \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@insurance_policy.pdf"

# 4. Poll for result
curl https://dpi-dev.tanuh.ai/pdf2nhcx/task-result/{task_id}

Concurrency & Deployment

            
The platform handles concurrent document processing through Celery distributed task queues backed by Redis.


            

                
SettingValueEffect

                
                    
Worker replicas33 independent Celery worker containers

                    
Concurrency per worker44 prefork processes per container

                    
Total slots1212 PDFs can process simultaneously

                    
Task time limit1800s (30min)Hard kill after 30 minutes

                
            


            

                Bottleneck: Vertex AI Rate Limits
                The actual throughput ceiling is Vertex AI API quotas, not infrastructure. Each PDF requires ~15 LLM calls.
