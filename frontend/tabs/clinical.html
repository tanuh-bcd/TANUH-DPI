Setup Steps (All Platforms)

1. Install Ollama

Download from ollama.com/download
Linux: curl -fsSL https://ollama.com/install.sh | sh

2. Pull the LLM Model (~16 GB download)

ollama pull gemma4:26b
Requires NVIDIA GPU with 16 GB+ VRAM, or Apple Silicon with 16 GB+ unified memory.

3. Unzip & Make Executable

Unzip the downloaded file. On Linux/Mac run: chmod +x nhcx-extract*
Linux requires sudo apt install libfuse2 if not already installed.

4. Run ABDM Clinical Extraction

./nhcx-extract abdm report.pdf -o result.json -m gemma4:26b

Use check first to verify setup: ./nhcx-extract check -m gemma4:26b

Privacy: All data stays on your machine. Zero cloud API calls. The output JSON contains a complete FHIR R4 Bundle.

 Overview

    
DPI Tanuh AI is a Digital Public Infrastructure platform developed by Tanuh AI-Center of Excellence at IISc Bangalore. The pdf2abdm service automates the conversion of unstructured clinical documents (discharge summaries, diagnostic reports) into machine-readable FHIR R4 DocumentBundles compliant with India's Ayushman Bharat Digital Mission (ABDM) standards.

    

        
FeatureDescription

        
            
Document IntelligenceAutomatic classification of clinical documents with two-tier validation gate

            
Multi-Patient SupportLab batch PDFs automatically split by patient fingerprint (Age/Sex + Collection Date)

            
FHIR R4 ComplianceGenerated bundles conform to ABDM StructureDefinition profiles

            
LangGraph OrchestrationDependency-aware extraction using topological sort for correct resource ordering

            
Async ProcessingCelery + Redis task queue supports up to 12 concurrent PDF processing slots

 ABDM Clinical Pipeline

    
The pdf2abdm service converts clinical medical documents into FHIR R4 DocumentBundles conforming to Ayushman Bharat Digital Mission profiles.


    
Supported Document Types

    

        
ArtifactDescriptionMandatory Resources

        
            
DischargeSummaryRecordInpatient discharge documentsPatient, Encounter, Practitioner, Organization, Condition, Procedure, Observation, DocumentReference

            
DiagnosticReportRecordLab and imaging reportsPatient, Practitioner, Organization, DiagnosticReportLab, Observation, DocumentReference

        
    


    
Pipeline Steps

    

        

            1. Multi-Engine OCR
            
PDF text extraction via 6-engine waterfall. Scanned documents trigger VLM/Tesseract fallback.

        

        

            2. Patient Grouping
            
For batch lab PDFs: pages grouped by Age/Sex + Collection Date fingerprint.

        

        

            3. Document Classification
            
LLM-based classifier selects clinical artifact type and identifies FHIR resources.

        

        

            4. LangGraph Extraction
            
Dependency-resolved workflow extracts each FHIR resource using rulebook-driven prompts.

        

        

            5. Bundle Assembly
            
Resources assembled into FHIR DocumentBundle with proper fullUrl references, PDF embedded as base64.

        

    


    
Multi-Patient Handling

    
Lab batch PDFs (e.g., multiple patients' blood work in one PDF) are detected by Age/Sex and Collection Date fields. Each patient group produces an independent FHIR bundle. Non-batch documents (discharge summaries) are always single-patient.

    

        Document Gate: Non-clinical documents (insurance PDFs, unrelated files) are rejected with HTTP 422 before any LLM processing occurs, saving compute resources.
    







    
 OCR Engine Waterfall

    
6-engine waterfall strategy to maximize text extraction quality across diverse PDF types (digital, scanned, handwritten, complex layouts).

    

        
#EngineTechnologyBest ForFallback Trigger

        
            
1PyPDFNative PDF text layerClean digital PDFs<50 chars/page

            
2Chandra2VLM 4B (datalab/chandra2)Complex layouts, handwritingGPU unavailable or model error

            
3DoclingIBM Structural ParserTables, structured documentsParse failure or empty output

            
4LightOnPyMuPDF4LLMFast digital extractionEmpty or insufficient text

            
5SuryaTransformer OCR90+ languages, 2x resolutionLow confidence scores

            
6ChandraTesseract (300 DPI)Last resort, any documentFinal fallback (always succeeds)

        
    

    

        Design Rationale: Fast path first (PyPDF for digital PDFs), VLM for quality (Chandra2 understands layout semantically), structural parsing (Docling preserves tables), and Tesseract as final fallback ensures every PDF produces output. All engines output through a common normalizer generating uniform Markdown.

 Validation & Sanitization

    
LLM-generated FHIR resources undergo multi-layer validation and correction via sanitize_fhir_resource() before inclusion in the final bundle.

    

        
IssueFix Applied

        
            
Wrong resourceTypeCorrected to match expected type

            
Missing required fieldsPopulated with sensible defaults

            
Invalid code systemsReplaced with correct canonical URLs

            
Broken internal referencesResolved using ID registry

            
Incorrect date formatsNormalized to FHIR dateTime format

            
Hallucinated extensionsRemoved if not in profile

        
    

    
Each resource uses a StructureDefinition JSON (rulebook) specifying required vs. optional fields (cardinality), allowed value sets and code systems, data type constraints, and slicing rules for complex types.
