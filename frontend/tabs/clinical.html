Setup Steps (All Platforms)

1. Install Ollama

Download from ollama.com/download
Linux: curl -fsSL https://ollama.com/install.sh | sh

2. Pull the LLM Model (~16 GB download)

ollama pull gemma4:26b
Requires NVIDIA GPU with 16 GB+ VRAM, or Apple Silicon with 16 GB+ unified memory.

3. Unzip & Make Executable

Unzip the downloaded file. On Linux/Mac run: chmod +x nhcx-extract*
Linux requires sudo apt install libfuse2 if not already installed.

4. Run ABDM Clinical Extraction

./nhcx-extract abdm report.pdf -o result.json -m gemma4:26b

Use check first to verify setup: ./nhcx-extract check -m gemma4:26b

Privacy: All data stays on your machine. Zero cloud API calls. The output JSON contains a complete FHIR R4 Bundle.

ABDM Clinical Pipeline

                
Automates conversion of unstructured clinical documents into machine-readable FHIR R4 DocumentBundles compliant with India's Ayushman Bharat Digital Mission (ABDM) standards.

            


            
Key Capabilities

            

                
FeatureDescription

                
                    
Document IntelligenceAutomatic classification of clinical documents with two-tier validation gate

                    
Multi-Patient SupportLab batch PDFs automatically split by patient fingerprint (Age/Sex + Collection Date)

How to Use

            
Follow these three steps to convert a clinical PDF into an ABDM-compliant FHIR bundle.


            

                

                    
1. Upload PDF

                    
Click "Choose PDF file..." and select a clinical document (discharge summary or diagnostic report). Maximum file size is 25 MB. Sample PDFs are provided above the upload area.

                

                

                    
2. Process Document

                    
Click "Process clinical document". The AI pipeline extracts text via OCR, classifies the document type, and generates FHIR resources. Processing takes 3–5 minutes. Do not close or refresh the tab.

                

                

                    
3. Download FHIR Bundle

                    
Once complete, the ABDM-compliant FHIR JSON appears in the output panel. Use Copy JSON or Download JSON to save. You can validate the bundle using the Validate FHIR R4 button.

                

            


            

                Privacy Protocol
                This is a non-persistent service. Your data is processed in real-time and purged automatically after use. No data is stored. Kindly download your JSON bundles to preserve your results.

Request Flow

            

                
1. Client Upload
PDF uploaded via frontend or API. Apache proxies to correct backend service.

                
2. Document Validation Gate
Two-tier classifier (keyword heuristic + LLM fallback) rejects misrouted documents.

                
3. OCR Extraction
6-engine waterfall extracts text with intelligent fallback for scanned/complex PDFs.

                
4. LLM-Powered FHIR Extraction
LangGraph StateGraph orchestrates Gemma 4 26B calls in dependency order.

                
5. Post-Processing & Validation
Sanitization fixes common LLM hallucinations, bundle assembly, PDF embedding.

                
6. Result Delivery
FHIR bundle stored in Redis (24h TTL)

            

        


        
        

            
ABDM Clinical Pipeline

            
The pdf2abdm service converts clinical medical documents into FHIR R4 DocumentBundles conforming to Ayushman Bharat Digital Mission profiles.


            
Supported Document Types

            

                
ArtifactDescriptionMandatory Resources

                
                    
DischargeSummaryRecordInpatient discharge documentsPatient, Encounter, Practitioner, Organization, Condition, Procedure, Observation, DocumentReference

                    
DiagnosticReportRecordLab and imaging reportsPatient, Practitioner, Organization, DiagnosticReportLab, Observation, DocumentReference

                
            


            
Pipeline Steps

            

                
1. Multi-Engine OCR
PDF text extraction via 6-engine waterfall. Scanned documents automatically trigger VLM/Tesseract fallback.

                
2. Patient Grouping (Lab Reports)
For diagnostic batch PDFs: pages grouped by Age/Sex + Collection Date fingerprint. Discharge summaries bypass this step.

                
3. Document Classification
LLM-based classifier selects clinical artifact type and identifies additional FHIR resources present in the text.

                
4. LangGraph Extraction
Dependency-resolved workflow extracts each FHIR resource using rulebook-driven prompts. Cross-references maintained via ID registry.

                
5. Bundle Assembly
Resources assembled into FHIR DocumentBundle with proper fullUrl references, PDF binary embedded as base64.

            


            
Multi-Patient Handling

            
Lab batch PDFs (e.g., multiple patients' blood work in one PDF) are detected by the presence of Age/Sex and Collection Date fields. When detected:

            
    
                
  • Pages are grouped by patient fingerprint
    • 
                
  • Each patient group produces an independent FHIR bundle
    • 
                
  • Non-batch documents (discharge summaries) are always treated as single-patient
    • 
            


            

                Document Gate
                Non-clinical documents (insurance PDFs, unrelated files) are rejected with HTTP 422 before any LLM processing occurs, saving compute resources.
            

        


        
        

            
OCR Engine Waterfall

            
The platform employs a 6-engine waterfall strategy to maximize text extraction quality across diverse PDF types (digital, scanned, handwritten, complex layouts).


            
Engine Priority Order

            

                
#EngineTechnologyBest ForFallback Trigger

                
                    
1PyPDFNative PDF text layerClean digital PDFs<50 chars/page (scanned)

                    
2Chandra2VLM 4B (datalab/chandra2)Complex layouts, handwritingGPU unavailable or error

                    
3DoclingIBM Structural ParserTables, structured docsParse failure or empty

                    
4LightOnPyMuPDF4LLMFast digital extractionEmpty or insufficient text

                    
5SuryaTransformer OCR90+ languages, 2x resolutionLow confidence scores

                    
6ChandraTesseract (300 DPI)Last resort, any documentFinal fallback (always succeeds)

                
            


            
Design Rationale

            
    
                
  • Fast path first: PyPDF is near-instant for digital PDFs (majority of clinical documents)
    • 
                
  • VLM for quality: Chandra2 (4B parameter model) understands document layout semantically
    • 
                
  • Structural parsing: Docling preserves tables and hierarchical structure
    • 
                
  • Never fail: Tesseract as final fallback ensures every PDF produces some output
    • 
            


            

                GPU Optimization
                Chandra2 (VLM) requires 8–12 GB VRAM. When no GPU is available, the waterfall automatically skips to Docling, maintaining service availability without hardware dependency.

LangGraph Workflow

            
    
                
  • Shared state: AgentState TypedDict carries text, ID registry, and extracted resources
    • 
                
  • Node factory: create_clinical_node() generates a node for each resource type
    • 
                
  • Topological ordering: Dependencies resolved automatically (Organization before Encounter)
    • 
                
  • Assembly node: Final node collects all resources into ordered DocumentBundle
    •

Generated resources conform to India-specific FHIR R4 profiles defined by NRCeS (National Resource Centre for EHR Standards).

ProcedureABDM ProcedureSurgical/medical procedures performed

Validation & Sanitization

            
LLM-generated FHIR resources undergo multi-layer validation and correction before inclusion in the final bundle.


            
Sanitization Pipeline

            
The sanitize_fhir_resource() function addresses common LLM hallucinations:

            

                
IssueFix Applied

                
                    
Wrong resourceTypeCorrected to match expected type

                    
Missing required fieldsPopulated with sensible defaults

                    
Invalid code systemsReplaced with correct canonical URLs

                    
Incorrect date formatsNormalized to FHIR dateTime format

                
            


            
Rulebook Enforcement

            
    
                
  • Required vs. optional fields (cardinality 1..1 vs 0..*)
    • 
                
  • Allowed value sets and code systems
    • 
                
  • Data type constraints (string, dateTime, Reference, etc.)
    • 
                
  • Slicing rules for complex types
    • 
            


            

                40+ Rulebooks
                The pdf2abdm/rulebooks_updated/ directory contains StructureDefinition JSONs for every supported ABDM resource type, loaded dynamically based on document classification.

Quick Start (curl)

            
# 1. Get a token
TOKEN=$(curl -s -X POST https://dpi-dev.tanuh.ai/pdf2abdm/api/token \
  -H "Content-Type: application/json" \
  -d '{"name":"Your Name","email":"you@example.com"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

# 2. Synchronous conversion (waits for result)
curl -X POST https://dpi-dev.tanuh.ai/pdf2abdm \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@discharge_summary.pdf"

# 3. Asynchronous (returns task_id immediately)
curl -X POST https://dpi-dev.tanuh.ai/pdf2abdm/submit \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@discharge_summary.pdf"

# 4. Poll for result
curl https://dpi-dev.tanuh.ai/pdf2abdm/task-result/{task_id}


            
Response Format

            
{
  "status": "completed",
  "bundles": [
    { "resourceType": "Bundle", "type": "document", ... }
  ],
  "doc_type": "DischargeSummaryRecord",
  "patient_count": 1,
  "model_used": "gemma4"
}


            

                Document Validation

Concurrency & Deployment

            
The platform handles concurrent document processing through Celery distributed task queues backed by Redis.


            
Default Configuration

            

                
SettingValueEffect

                
                    
Worker replicas33 independent Celery worker containers

                    
Concurrency per worker44 prefork processes per container

                    
Total slots1212 PDFs can process simultaneously

                    
Task time limit1800s (30min)Hard kill after 30 minutes

                    
Soft time limit1740s (29min)SoftTimeLimitExceeded raised at 29 min

                
            


            
Progress Tracking

            

                
StepProgressDescription

                
                    
OCR15%Text extraction in progress

                    
Classifying document30%Resource type identification

                    
LLM Extraction50%FHIR resource generation

                    
Storing results95%Redis caching

                    
Completed100%Ready for retrieval

                
            


            

                Bottleneck: Vertex AI Rate Limits
                The actual throughput ceiling is not infrastructure but Vertex AI API quotas. Each PDF requires ~10 LLM calls. With default quotas, effective throughput is approximately 3–4 concurrent conversions regardless of worker count.
