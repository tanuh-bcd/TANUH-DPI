DPI Tanuh AI Documentation

A comprehensive AI-powered platform for converting medical PDF documents into HL7 FHIR R4 compliant bundles, supporting both ABDM (clinical) and NHCX (insurance) Indian healthcare interoperability standards.

What is DPI Tanuh AI?

DPI Tanuh AI is a Digital Public Infrastructure platform developed by Tanuh AI-Center of Excellence at IISc Bangalore that automates the conversion of unstructured medical documents (insurance policies, discharge summaries, diagnostic reports) into machine-readable FHIR bundles compliant with India's national health data exchange standards.

ABDM Clinical Pipeline

Discharge summaries and diagnostic reports to ABDM-compliant FHIR DocumentBundles.

NHCX Insurance Pipeline

Insurance policies and claims to NHCX-compliant FHIR InsurancePlanBundle

Multi-Engine OCR

6-engine waterfall for maximum text extraction quality

REST API

Synchronous and asynchronous endpoints for integration

Forgery Detection

Classical CV forensics: detect tampering, splicing, and forgery in clinical documents.

ABDM Clinical Pipeline

The pdf2abdm service converts clinical medical documents (discharge summaries, diagnostic reports) into FHIR R4 DocumentBundles conforming to Ayushman Bharat Digital Mission profiles.

Supported Document Types

ArtifactDescriptionMandatory Resources
DischargeSummaryRecord Inpatient discharge documents Patient, Encounter, Practitioner, Organization, Condition, Procedure, Observation, DocumentReference
DiagnosticReportRecord Lab and imaging reports Patient, Practitioner, Organization, DiagnosticReportLab, Observation, DocumentReference

Pipeline Flow

1. Multi-Engine OCR

PDF text extraction via 6-engine waterfall. Scanned documents automatically trigger VLM/Tesseract fallback.

2. Patient Grouping (Lab Reports)

For diagnostic batch PDFs: pages grouped by Age/Sex + Collection Date fingerprint. Discharge summaries bypass this step.

3. Document Classification

LLM-based classifier selects clinical artifact type and identifies additional FHIR resources present in the text.

4. LangGraph Extraction

Dependency-resolved workflow extracts each FHIR resource using rulebook-driven prompts. Cross-references maintained via ID registry.

5. Bundle Assembly

Resources assembled into FHIR DocumentBundle with proper fullUrl references, PDF binary embedded as base64.

Multi-Patient Handling

Lab batch PDFs (e.g., multiple patients' blood work in one PDF) are detected by the presence of Age/Sex and Collection Date fields. When detected:

  • Pages are grouped by patient fingerprint
  • Each patient group produces an independent FHIR bundle
  • Non-batch documents (discharge summaries) are always treated as single-patient
Document Gate Non-clinical documents (insurance PDFs, unrelated files) are rejected with HTTP 422 before any LLM processing occurs, saving compute resources.

NHCX Insurance Pipeline

The pdf2nhcx service converts insurance documents into FHIR R4 InsurancePlanBundle resources conforming to National Health Claims Exchange standards. All insurance document types are converted into this single bundle format.

Input & Output

The pipeline accepts any insurance-related PDF document (health insurance policies, claims, etc.) as input. The output is always a single InsurancePlanBundle — a FHIR Bundle of type collection containing InsurancePlan, Organization, Condition, DocumentReference, and other relevant resources extracted from the document.

Resource Dependency Graph

NHCX resources have strict ordering requirements. The pipeline uses topological sorting to ensure dependencies are resolved before dependent resources are extracted:

Organization ──────────┐ v InsurancePlan ─────────┤ v InsurancePlanBundle ───┘ Independent: Binary, DocumentReference, Condition, Patient, Practitioner

Pipeline Steps

1. OCR + Distillation

Multi-engine OCR extracts raw text, then 8 LLM calls distill structured information from the entire text.

2. Document Type Gate

Classifier validates the document is actually an insurance document. Clinical docs are rejected with a redirect message.

3. NHCX Resource Selection

LLM identifies which insurance-specific FHIR resources are present in the document.

4. LangGraph Extraction (Dependency-Ordered)

Each resource extracted via dedicated LLM call with NHCX rulebook (StructureDefinition JSON) loaded into prompt context.

5. Sanitization & Assembly

sanitize_fhir_resource() fixes hallucinated types, missing required fields, invalid code systems. Bundle assembled with InsurancePlan first.

6. PDF Embedding & Caching

Original PDF embedded as base64 Binary resource. Final bundle cached in Redis (24h TTL).

Rulebook-Driven Extraction

Each FHIR resource type has a corresponding StructureDefinition JSON file (40+ rulebooks) loaded into the LLM prompt. This ensures generated resources conform to the exact field constraints, cardinality rules, and terminology bindings defined by NHCX.

Cross-Reference Integrity An id_registry dictionary tracks generated resource IDs across the workflow, ensuring that references (e.g., InsurancePlan referencing Organization) use correct UUIDs.

OCR Engine Waterfall

The platform employs a 6-engine waterfall strategy to maximize text extraction quality across diverse PDF types (digital, scanned, handwritten, complex layouts).

Engine Priority Order

#EngineTechnologyBest ForFallback Trigger
1PyPDFNative PDF text layerClean digital PDFs<50 chars/page (scanned PDF)
2Chandra2VLM 4B (datalab/chandra2)Complex layouts, handwritingGPU unavailable or model error
3DoclingIBM Structural ParserTables, structured documentsParse failure or empty output
4LightOnPyMuPDF4LLMFast digital extractionEmpty or insufficient text
5SuryaTransformer OCR90+ languages, 2x resolutionLow confidence scores
6ChandraTesseract (300 DPI)Last resort, any documentFinal fallback (always succeeds)

Design Rationale

  • Fast path first: PyPDF is near-instant for digital PDFs (majority of insurance documents)
  • VLM for quality: Chandra2 (4B parameter model) understands document layout semantically
  • Structural parsing: Docling preserves tables and hierarchical structure
  • Never fail: Tesseract as final fallback ensures every PDF produces some output

Output Normalization

All engines produce output through a common normalizer that generates uniform Markdown with:

  • YAML front-matter (engine used, page count, confidence)
  • <!-- PAGE N --> markers for page boundaries
  • Consistent heading levels and table formatting
GPU Optimization Chandra2 (VLM) requires 8-12GB VRAM. When no GPU is available, the waterfall automatically skips to Docling, maintaining service availability without hardware dependency.

LangGraph Workflow

The extraction workflow is built as a LangGraph StateGraph with:

  • Shared state: AgentState TypedDict carries text, ID registry, and extracted resources
  • Node factory: create_insurance_node() generates a node for each resource type
  • Topological ordering: Dependencies resolved automatically (Organization before InsurancePlan)
  • Assembly node: Final node collects all resources into ordered bundle

Validation & Sanitization

LLM-generated FHIR resources undergo multi-layer validation and correction before inclusion in the final bundle.

Sanitization Pipeline

The sanitize_fhir_resource() function addresses common LLM hallucinations:

IssueFix Applied
Wrong resourceTypeCorrected to match expected type
Missing required fieldsPopulated with sensible defaults
Invalid code systemsReplaced with correct canonical URLs
Broken internal referencesResolved using ID registry
Incorrect date formatsNormalized to FHIR dateTime format
Hallucinated extensionsRemoved if not in profile

Rulebook Enforcement

Each resource extraction uses a StructureDefinition JSON (rulebook) that specifies:

  • Required vs. optional fields (cardinality 1..1 vs 0..*)
  • Allowed value sets and code systems
  • Data type constraints (string, dateTime, Reference, etc.)
  • Slicing rules for complex types
40+ Rulebooks The pdf2nhcx/rulebooks_updated/ directory contains StructureDefinition JSONs for every supported NHCX resource type, loaded dynamically based on document classification.

Forgery Detection — Forgensic

Forgensic is a classical computer-vision forensic analysis system developed by TANUH at IISc Bengaluru for the AB PMJAY Auto-Adjudication Hackathon. It detects tampering, splicing, copy-paste, and manipulation in clinical and administrative documents without any machine-learning inference. The pipeline is deterministic, explainable, and runs entirely on CPU.

No AI / No GPU required Forgensic uses only classical OpenCV operations and PyMuPDF. Unlike the PDF2ABDM and PDF2NHCX pipelines which rely on Vertex AI Gemma 4, Forgensic has zero LLM dependency and can run fully offline.

Forgensic — CV Detection Pipeline

Every uploaded page passes through the following sequence of classical computer vision checks. The pipeline behaviour is controlled by the PIPELINE_PRESET environment variable.

Pipeline Stages

1. Image Pre-processing & Page Rendering

PDF pages rendered to high-resolution JPG via PyMuPDF (fitz). Single image files are loaded directly. DPI scaling is applied for detection accuracy on lower-resolution scans.

2. Copy-Paste Detection (C1)

ORB keypoint extraction + FLANN-based KNN matching to find duplicate regions within the same page. Block-level perceptual hashing catches compressed re-saves. Flagged region bounding boxes are stored for the frontend overlay.

3. Overwriting Detection (C2)

Edge density and stroke-width analysis within text-line bounding boxes. Ink inconsistency is scored against the surrounding line's baseline stroke profile. High-contrast edge clusters in unexpected positions indicate overwritten content.

4. Added Content Detection (C3)

Stamp and signature heuristics: HSV colour saturation thresholding, contour shape analysis, and aspect-ratio filtering. Catches rubber-stamp marks, signature overlays, and pasted image regions that have different colour profiles from the document background.

5. Erasure Detection (C4)

Laplacian noise variance and gradient smoothness analysis in regions adjacent to text lines. Abnormally smooth zones (white-out, digital erase, paint-bucket fills) produce near-zero Laplacian variance — a strong erasure signal.

6. Document Merge Detection (C5)

The page is divided into horizontal bands. Laplacian variance (noise fingerprint) is computed per band. A statistically significant jump between the header band and the body band indicates the page was assembled from documents printed on different printers or scanned at different qualities.

7. Watermark Removal Detection (C6)

FFT frequency domain analysis detects residual ghost patterns left by watermark removal tools. Background uniformity scoring identifies pixel-level artifacts that betray post-processing.

8. Irregular Spacing Detection (C7)

Word-level bounding boxes from the Tesseract OCR pass are used to measure inter-token distances on each line. Statistical outlier detection (IQR method) flags lines where a gap is abnormally wider than the document's typical spacing — a sign of inserted or deleted words.

9. AI-Generated / AI-Edited Flags (C8, C9)

Texture uniformity analysis (too-smooth gradients), font embedding entropy scoring, and micro-print rendering anomaly detection are combined to flag documents that appear fully (C8) or partially (C9) AI-generated.

10. Annotated Preview Generation

All detected bounding boxes are rendered onto the page image using Pillow, colour-coded by category (e.g., red for C1 copy-paste, orange for C2 overwrite). Saved as preview_{page}.jpg and registered in the Redis file_map for serving via GET /files/*.

Python: End-to-End Example

import time, requests

BASE = "https://dpi-dev.tanuh.ai/forgensic"

token = requests.post(
    f"{BASE}/api/token",
    json={"name": "Dev", "email": "dev@tanuh.ai"},
).json()["access_token"]

headers = {"Authorization": f"Bearer {token}"}

with open("document.pdf", "rb") as f:
    job_id = requests.post(
        f"{BASE}/jobs",
        files={"file": f},
        data={"ocr_enabled": "true"},
        headers=headers,
    ).json()["job_id"]

# Poll until done
for _ in range(120):
    s = requests.get(f"{BASE}/jobs/{job_id}", headers=headers).json()
    if s["status"] in ("complete", "error"):
        break
    time.sleep(2)

# Fetch results
results = requests.get(f"{BASE}/jobs/{job_id}/results", headers=headers).json()
print(results["category_summary"])
print(results["findings_summary"]["summary_text"])

# Download annotated preview image
for page in results["pages"]:
    if page["preview_url"]:
        img = requests.get(f"{BASE}{page['preview_url']}", headers=headers)
        with open(f"preview_page_{page['page_number']}.jpg", "wb") as f:
            f.write(img.content)

Forgensic — Deployment & Configuration

Docker Compose (Recommended — Linux / GCP VM)

The docker-compose.yml in the project root launches Redis, the FastAPI server, and the Celery worker together. On Linux, Celery automatically uses the prefork pool for full multi-process isolation.

# Clone the repository
git clone https://github.com/tanuh-bcd/TANUH-DPI.git
cd TANUH-DPI

# Set secrets
cat <<EOF > .env
FORGENSIC_SECRET_KEY=your-secure-random-key
REDIS_URL=redis://redis:6379/0
OCR_ENABLED=true
PIPELINE_PRESET=super_loose
EOF

# Start all services
docker-compose up -d --build

# Verify
docker-compose ps
docker-compose logs -f forgensic-worker
GCP VM sizing Use e2-standard-4 (4 vCPU, 16 GB) or larger. Each Celery prefork worker uses one CPU core. The throughput scales linearly: 4 cores = ~4 concurrent pages/second.

Windows Bare-Metal (Development)

Windows does not support Celery's prefork pool. Use --pool=solo and launch multiple worker processes manually to simulate concurrency.

# 1. Start Redis via WSL
wsl -e bash -c "sudo service redis-server start"

# 2. Environment variables (PowerShell)
$env:PYTHONPATH        = "C:\path\to\TANUH-DPI"
$env:REDIS_URL         = "redis://localhost:6379/0"
$env:DATA_DIR          = "$env:TEMP\forgensic_data"
$env:OCR_ENABLED       = "true"
$env:PIPELINE_PRESET   = "super_loose"
$env:FORGENSIC_SECRET_KEY = "dev-secret-local"

# 3. Start API (Terminal 1)
uvicorn forgensic.app.main:app --host 0.0.0.0 --port 8004 --reload

# 4. Start Worker (Terminal 2)
celery -A forgensic.celery_app worker --loglevel=info --pool=solo -Q forgensic -n worker1@%h

Shared DATA_DIR is critical The FastAPI container saves uploaded files to DATA_DIR. The Celery worker reads them from the same path. In Docker, both containers must mount the same volume. In bare-metal deployments, both processes must share the same filesystem path.

Privacy & Data Handling

  • All processing is local — no external API calls are made
  • Uploaded files are stored in DATA_DIR during processing only
  • Annotated preview images are cached for the job TTL window, then deleted
  • No document content is written to any database or persistent store
  • Jobs are fully purged from Redis after JOB_TTL_SECONDS
